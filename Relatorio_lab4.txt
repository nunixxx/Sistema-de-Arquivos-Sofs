--- 1. STATUS DE IMPLEMENTAÇÃO ---

* sofs_create / sofs_delete: 

O sofs_create verifica se o arquivo já existe, em caso positivo, ele reaproveita o i-node e zera seus blocos de dados. Já o sofs_delete primeiro verifica
o RefCounter, com o objetivo de respeitar hard-links, e assim que o contador chega a zero, os blocos e o i-node são liberados.

* sofs_open / sofs_close: 

O sofs_open busca o espaço vazio na tabela global de arquivos e endereça o arquivo para esse local (respeitando juntamente o limite de 10 arquivos abertos
simultaneamente). Além disso, a função verifica o uso de softlinks, limitando a busca a uma profundidade de 8 iterações, assim não há a chance de loops 
infinitos (no caso de arquivos mutuamente se apontarem).A sofs_close verifica o handle do arquivo a ser fechado e libera o espaço na tabela global, 
substituindo o valor por -1.

* sofs_read / sofs_write: 

O soft_read é capaz de lidar com a leitura de arquivos em fronteira de bloco de disco,manipulando o offset, fazendo a transferência de dados por meio de
camadas de blocos. O sofs_write utiliza de demandas do sistema para alocar dinamicamente espaço para a escrita de dados, o que também trata da quebra
de blocos.

* sofs_opendir / sofs_readdir/ sofs_closedir: 

O sofs_opendir é utilizado para abrir o diretório operando sobre o bloco de dados de i-node 0. A função de sofs_readir itera sobre as entradas do diretório
e filtra as entradas inválida e ignora entradas deletadas (TYPEVAL_INVALIDO), retornando as entradas válidas. A operação de sofs_closedir encerra a operação,
zerando as variáveis de controle.

* sofs_sln (softlink): 

O sofs_sln aloca um i-node do tipo TYPEVAL_LINK, juntamente com um bloco de dados correspondente. Em seguida, grava a string com o caminho do arquivo alvo
dentro desse bloco de dados.

* sofs_hln (hardlink): 

O sofs_hln utiliza resolve_name() para seguir softlinks recursivamente até o arquivo regular alvo, criando uma entrada no diretório raiz que aponta para 
o mesmo i-node e incrementando seu RefCounter.

--- 2. DECISÕES DE PROJETO ---

Tratamento de Links: Implementamos uma função chamada resolve_name() que é chamada ao abrir arquivos ou criar hard-links, com o intuito de resolver o caminho
final do arquivo. Além disso, no caso de links que apontam para links, determinamos uma variável MAX_LINK_DEPTH de valor 8, com o objetivo de evitar loops 
infinitos nesse cenário.


O diretório raiz: O espaço do diretório raiz é gerenciado dinamicamente. Se a função find_free_record() não encontrar slots com TYPEVAL_INVALIDO no bloco atual,
ela aloca novos blocos de dados para expandir a capacidade de armazenamento do diretório raiz.

Resolução de blocos: Utilizamos da função get_data_block() para lidar com a lógica de tradução de endereços lógicos para físicos, com o fim de facilitar
operações de escrita e leitura. Ela se utiliza de ponteiros diretos, indireção simples e indireção dupla para navegar os i-nodes. Ao receber flags de alocação
(durante escrita), a função se encarrega de inicilizar os índices correpondentes em disco.
